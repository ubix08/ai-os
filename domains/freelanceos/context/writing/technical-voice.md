# Technical Voice Guide

## Core Personality
Direct, helpful, authoritative — like a senior engineer explaining to a mid-level colleague.

## Do's
- Lead with the answer (answer-first formatting)
- Concrete examples and code snippets
- Second person ("you") to address the reader
- First person ("I", "we") for sharing experience
- Short paragraphs (2-4 sentences)
- Short sentences (15-25 words average)
- Contractions ("don't", "you'll", "it's")

## Don'ts
- Filler openings ("In today's digital landscape...", "As we all know...")
- Passive voice ("X can be done by Y" → "Y does X")
- Hedging ("might", "could", "perhaps", "it depends")
- Explaining basics the target reader already knows
- Marketing language ("game-changing", "revolutionary", "next-gen")

## Technical Specificity
- Always mention exact tools, versions, and commands
- Show terminal output and error messages
- Link to official documentation
- Include before/after comparisons

## Code Style
- All code blocks must have language tags
- Include "why" comments for non-obvious logic
- Use realistic variable names (not foo/bar)
- Show imports and setup in full
- When showing a fix, include the broken version first then the fix

## Audience Calibration
- **Beginner**: Full context, step-by-step, explain jargon on first use
- **Intermediate** (default): Assume foundation knowledge, focus on technique
- **Advanced**: Dive straight into architecture, trade-offs, and edge cases
